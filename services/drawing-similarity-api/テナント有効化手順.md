**新規ドキュメント（ID = 上のハッシュ値64文字）** を作成し、フィールドを設定:

## STEP 4.5: Firestore データベースIDの確認（重要）

Firestore のデータベースIDが `(default)`（括弧つき）でない場合、サーバーに
その名前を教える必要がある。コンソール → Firestore → データベースの詳細 →
「データベース ID」を確認：

- `(default)` → 何もしなくてよい（既定）
- `default` など名前付き → Cloud Run に環境変数を追加：
  ```sh
  gcloud run services update drawing-similarity-api \
    --update-env-vars FIRESTORE_DATABASE_ID=default \
    --region asia-northeast1 --project drawing-similarity-501101
  ```
  （`default` の部分を実際のデータベースIDに置き換える）

## つまずきポイント（2026-07-06 の実作業で全て踏んだ・要注意）

401 / 接続不可のとき、原因はほぼこの4つのどれか。上から順に潰す：

1. **CORS（キーありだけ失敗）**: 「x-api-key is not allowed by Access-Control-Allow-Headers」
   → デプロイ済みコードが古い。最新を再デプロイ（STEP 1）。
2. **Firestore DB名**: 既定DBが `(default)` でなく `default` などの名前付きだと、
   コードが `(default)` を見て 404→401。→ STEP 4.5 の `FIRESTORE_DATABASE_ID`。
3. **ハッシュ不一致**: ドキュメントID = `sha256(プラグインのキー)` が一致必須。
   `echo`（改行あり）で作るとズレる。**キーとハッシュは同時生成**が安全：
   ```sh
   NEWKEY=$(openssl rand -hex 20)
   echo "key=$NEWKEY  docId=$(printf %s "$NEWKEY" | sha256sum | cut -d' ' -f1)"
   ```
4. **サービス二重作成（最も気づきにくい）**: `gcloud run deploy` をサービス名なし/
   誤った場所で実行すると別サービスが増え、修正が別々の箱に散る。
   → **必ず `gcloud run deploy drawing-similarity-api ...` と名前を明示**。
   プラグインの apiBaseUrl が指す1つのサービスに、コード＋env＋権限を集約する。

**切り分けの鉄板コマンド**（プロジェクト共通なのでサービスに依らない）:
```sh
# サーバーと同じハッシュで、同じDBパスの実在確認（NOT_FOUND ならドキュメント側の問題）
TOKEN=$(gcloud auth print-access-token)
HASH=$(printf %s "プラグインのキー" | sha256sum | cut -d' ' -f1)
curl -sS -H "Authorization: Bearer $TOKEN" \
  "https://firestore.googleapis.com/v1/projects/drawing-similarity-501101/databases/default/documents/tenants/$HASH"
# サーバー経由（200 なら解決）
curl -sSi 'https://drawing-similarity-api-939943665629.asia-northeast1.run.app/tags?tenantId=yahatausa' \
  -H "X-API-Key: プラグインのキー" | head -1
```

**サービスアカウント権限**: 2024年以降作成のプロジェクトは、デフォルト compute SA に
Firestore 権限が無い。→ `roles/datastore.viewer` を付与：
```sh
gcloud projects add-iam-policy-binding drawing-similarity-501101 \
  --member="serviceAccount:939943665629-compute@developer.gserviceaccount.com" \
  --role="roles/datastore.viewer"
```

## A. サーバー再デプロイ（コードを更新するとき）

**必ずこの手順。** `git pull` だけ／env更新だけではコードは反映されない。

```sh
# 1) Cloud Shell で最新コードに確実に合わせる
cd ~/plugbits-plugins && git fetch origin && git reset --hard origin/main
git log -1 --oneline                      # 期待するコミットが出ることを確認

# 2) サービス名を明示してビルド＆デプロイ（Dockerfile のあるディレクトリで）

  --source . \
  --region asia-northeast1 --project drawing-similarity-501101
```

**厳守事項（実作業で踏んだ罠）:**
- 先頭の `drawing-similarity-api` を**必ず明示**。省くと別サービスが増え、修正が散る。
- 必ず `services/drawing-similarity-api`（Dockerfile のある場所）に `cd` してから `--source .`。
  ホーム直下だと Dockerfile が無くビルド失敗、または `Source code location` プロンプトが出る。
- `gcloud run services update --update-env-vars ...` は**イメージを再ビルドしない**。
  コードを変えたら必ず上の `deploy --source .` を実行する。
- 赤い `Regional Access Boundary ... Account not found` はアカウント側のノイズで無視してよい。
  最後に `Service [...] has been deployed and is serving 100 percent of traffic` が出れば成功。

## B. テナント発行（APIキー＋ハッシュID＋Firestoreドキュメント）

**キーとハッシュIDは必ず同時生成する**（別々に作ると改行混入等でズレて 401 になる）。

```sh
# キーと、そのSHA-256（＝FirestoreのドキュメントID）を同時に出す
NEWKEY=$(openssl rand -hex 20)
echo "APIキー（プラグインに貼る）    : $NEWKEY"
echo "ドキュメントID（Firestoreに貼る): $(printf %s "$NEWKEY" | sha256sum | cut -d' ' -f1)"
```

Firestore コンソール → `default` データベース → `tenants` コレクション → **ドキュメントを追加**:
- **ドキュメントID** = 上の「ドキュメントID」64文字を貼る（自動生成の20文字IDは消して上書き）
- フィールド:
  - `tenantId`（型 **string**） = そのテナントの kintone サブドメイン（例 `yahatausa`）
  - `active`（型 **boolean**） = `true`（文字列の "true" にしない）

プラグイン設定 → API Key に `NEWKEY` を貼って保存（再パック不要・設定変更のみ）。

**検証（ブラウザ不要・200 なら成功）:**
```sh
curl -sSi 'https://drawing-similarity-api-939943665629.asia-northeast1.run.app/tags?tenantId=そのサブドメイン' \
  -H "X-API-Key: 貼ったキー" | head -1
```

## C. テナントID（tenantId）の考え方

- `tenantId` は **kintone のサブドメイン**（`https://<sub>.kintone.com` の `<sub>`）。例: `yahatausa`。
- プラグインは検索/登録時にこの値を送るが、**認証ON時はサーバーが無視**し、
  API キーに紐づく Firestore ドキュメントの `tenantId` を正として上書きする（Phase 2）。
- したがって **Firestore ドキュメントの `tenantId` が実質の唯一の正**。ここを間違えると、
  そのテナントのデータが別テナント名で保存され検索に出なくなる。サブドメインと一致させること。
- Qdrant のデータ分離もこの `tenantId` で行われる。テナントを変えたら再インデックスが必要。

## 2社目テナントを追加するとき

1. 上の **B**（または **B'**）の手順で新キー＋ハッシュIDを発行し、`tenants` に
   ドキュメント作成（`tenantId` = 新社の kintone サブドメイン, `active` = true）。
2. 新社の kintone にプラグイン導入・設定（同じ Base URL＋新キー）→ 接続テスト。
3. **kintone接続について**: プラグインがブラウザのkintoneセッションでPDFを取得して
   `pdf_base64` としてサーバーに送る方式になったため、**別 kintone 環境の2社目でも
   キー発行のみ（上のSTEP 1・2）で動く**。Phase 3（テナント別 kintone 接続）は不要。
   サーバーの `KINTONE_BASE_URL` / `KINTONE_API_TOKEN` は旧プラグイン（fileKeyのみ送信）
   との互換用に残しているだけで、新プラグインは使わない。新プラグインの配布が
   全テナントに行き渡ったら、次のコマンドで削除してよい（`THUMB_TOKEN_SECRET` を
   別途設定している場合はそのまま影響を受けない）:
   ```sh
   gcloud run services update drawing-similarity-api \
     --remove-env-vars KINTONE_BASE_URL,KINTONE_API_TOKEN \
     --region asia-northeast1 --project drawing-similarity-501101
   ```
