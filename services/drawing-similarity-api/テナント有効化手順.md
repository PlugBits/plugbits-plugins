## STEP 3: Firestore テナントドキュメント整備（Phase 2 の要）

APIキーは既存のものを使ってよい。Cloud Shell で:

```sh
echo -n "実際のAPIキー" | sha256sum
```

Firestore コンソール → `tenants` コレクション →
**新規ドキュメント（ID = 上のハッシュ値64文字）** を作成し、フィールドを設定:

| フィールド | 型 | 値 |
|---|---|---|
| tenantId | string | yahatausa |
| active | boolean | true |

- **`tenantId` が無いと、サーバーはクライアント送信値にフォールバックし
  Phase 2 の「キー＝テナント」強制が効かない。必ず入れる。**
- 旧ドキュメント（生キーがIDのもの）は STEP 6 の動作確認が終わってから削除。

## STEP 4.5: Firestore データベースIDの確認（重要）

Firestore のデータベースIDが `(default)`（括弧つき）でない場合、サーバーに
その名前を教える必要がある。コンソール → Firestore → データベースの詳細 →
「データベース ID」を確認：

- `(default)` → 何もしなくてよい（既定）
- `default` など名前付き → Cloud Run に環境変数を追加：
  ```sh
  gcloud run services update drawing-similarity-api \
    --update-env-vars FIRESTORE_DATABASE_ID=default \
    --region asia-northeast1 --project drawing-similarity-501101
  ```
  （`default` の部分を実際のデータベースIDに置き換える）

## つまずきポイント（2026-07-06 の実作業で全て踏んだ・要注意）

401 / 接続不可のとき、原因はほぼこの4つのどれか。上から順に潰す：

1. **CORS（キーありだけ失敗）**: 「x-api-key is not allowed by Access-Control-Allow-Headers」
   → デプロイ済みコードが古い。最新を再デプロイ（STEP 1）。
2. **Firestore DB名**: 既定DBが `(default)` でなく `default` などの名前付きだと、
   コードが `(default)` を見て 404→401。→ STEP 4.5 の `FIRESTORE_DATABASE_ID`。
3. **ハッシュ不一致**: ドキュメントID = `sha256(プラグインのキー)` が一致必須。
   `echo`（改行あり）で作るとズレる。**キーとハッシュは同時生成**が安全：
   ```sh
   NEWKEY=$(openssl rand -hex 20)
   echo "key=$NEWKEY  docId=$(printf %s "$NEWKEY" | sha256sum | cut -d' ' -f1)"
   ```
4. **サービス二重作成（最も気づきにくい）**: `gcloud run deploy` をサービス名なし/
   誤った場所で実行すると別サービスが増え、修正が別々の箱に散る。
   → **必ず `gcloud run deploy drawing-similarity-api ...` と名前を明示**。
   プラグインの apiBaseUrl が指す1つのサービスに、コード＋env＋権限を集約する。

**切り分けの鉄板コマンド**（プロジェクト共通なのでサービスに依らない）:
```sh
# サーバーと同じハッシュで、同じDBパスの実在確認（NOT_FOUND ならドキュメント側の問題）
TOKEN=$(gcloud auth print-access-token)
HASH=$(printf %s "プラグインのキー" | sha256sum | cut -d' ' -f1)
curl -sS -H "Authorization: Bearer $TOKEN" \
  "https://firestore.googleapis.com/v1/projects/drawing-similarity-501101/databases/default/documents/tenants/$HASH"
# サーバー経由（200 なら解決）
curl -sSi 'https://drawing-similarity-api-939943665629.asia-northeast1.run.app/tags?tenantId=yahatausa' \
  -H "X-API-Key: プラグインのキー" | head -1
```

**サービスアカウント権限**: 2024年以降作成のプロジェクトは、デフォルト compute SA に
Firestore 権限が無い。→ `roles/datastore.viewer` を付与：
```sh
gcloud projects add-iam-policy-binding drawing-similarity-501101 \
  --member="serviceAccount:939943665629-compute@developer.gserviceaccount.com" \
  --role="roles/datastore.viewer"
```

## STEP 6: 動作確認チェックリスト（認証ON後の初回）

3. 類似検索で結果が返ることを確認

## 2社目テナントを追加するとき

1. 新しいAPIキーを発行（ランダム32文字以上）
2. `echo -n "新キー" | sha256sum` → ハッシュIDで `tenants` にドキュメント作成
   （tenantId = 新社の kintone サブドメイン, active = true）
3. 新社の kintone にプラグイン導入・設定（同じ Base URL＋新キー）
4. 注意: kintone 接続情報（KINTONE_BASE_URL/API_TOKEN）は現状サーバーに1組のみ。
   **別 kintone 環境の2社目を入れる前に Phase 3（テナント別 kintone 接続）が必要。**
