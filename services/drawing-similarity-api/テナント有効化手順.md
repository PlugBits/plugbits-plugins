# テナント認証 有効化手順（Phase 2 + セキュリティ強化の反映）

最終更新: 2026-07-02
対象コミット: 2548c70（Phase 2）/ 506831a（UI刷新）/ a89c1f4（セキュリティ強化）

この順番で実施すること。STEP 3 を飛ばすと Phase 2 の分離保証が効かないまま動く。

---

## STEP 1: サーバーを最新コードでデプロイ

Cloud Shell で:

```sh
cd plugbits-plugins && git pull origin main
cd services/drawing-similarity-api
gcloud run deploy drawing-similarity-api \
  --source . \
  --region asia-northeast1 \
  --project drawing-similarity-501101
```

**確認**: 表示された Service URL で `/health` を開き、`runtime` が見えること
（この時点では認証OFFなので全情報が見える。これが正常）。

## STEP 2: Cloud Run の運用設定（1回だけ）

暴走時の課金キャップと同時実行の制限:

```sh
gcloud run services update drawing-similarity-api \
  --max-instances=3 --concurrency=4 \
  --region asia-northeast1 --project drawing-similarity-501101
```

**確認（課金モード）**: コンソール → サービス → リビジョン詳細 →
「CPU の割り当て」が「**リクエストの処理中にのみCPUを割り当てる**」であること。
これなら min=1 でもアイドル料金のみ（1 vCPU/4 GiB で月 $25〜35 目安）。

## STEP 3: Firestore テナントドキュメント整備（Phase 2 の要）

APIキーは既存のものを使ってよい。Cloud Shell で:

```sh
echo -n "実際のAPIキー" | sha256sum
```

Firestore コンソール → `tenants` コレクション →
**新規ドキュメント（ID = 上のハッシュ値64文字）** を作成し、フィールドを設定:

| フィールド | 型 | 値 |
|---|---|---|
| tenantId | string | yahatausa |
| active | boolean | true |

- **`tenantId` が無いと、サーバーはクライアント送信値にフォールバックし
  Phase 2 の「キー＝テナント」強制が効かない。必ず入れる。**
- 旧ドキュメント（生キーがIDのもの）は STEP 6 の動作確認が終わってから削除。

## STEP 4: プラグイン更新

1. ローカルで再パック（署名鍵 .ppk 使用）→ kintone システム管理 → プラグイン → 読み込み（更新）
2. アプリのプラグイン設定を開く:
   - API Base URL = STEP 1 の Service URL
   - API Key = STEP 3 で使ったキー
   - **「接続テスト」を押して「✓ 接続OK」を確認**
   - 保存

## STEP 5: テナント認証を有効化

```sh
gcloud run services update drawing-similarity-api \
  --update-env-vars TENANT_AUTH_ENABLED=true \
  --region asia-northeast1 --project drawing-similarity-501101
```

## STEP 6: 動作確認チェックリスト（認証ON後の初回）

- [ ] 設定画面「接続テスト」→ ✓ 接続OK（runtime情報が表示される）
- [ ] キー無しで `/health` を開く → `{ok, service}` だけになっている（情報最小化の確認）
- [ ] 詳細画面「類似図面検索」→ 結果表示・**サムネイル画像が出る**（HMACトークンの確認）
- [ ] 一覧「類似検索（アップロード）」→ 結果表示（mode: qdrant-…）
- [ ] （任意）設定でキーを一時的に空にして検索 → 「APIキーが正しくありません」表示 → 戻す

## STEP 7: 既存インデックスの再構築（必須・1回だけ）

Phase 2 で Qdrant のポイントID体系が変わったため（tenantId+recordId のハッシュ化）、
既存データは再登録が必要。

1. （推奨）Qdrant Cloud コンソール → コレクション
   `drawing_similarity_dinov2_base` を**削除**（旧ポイントの掃除。次回登録時に自動再作成）
2. kintone 一覧画面 → 「**一括図面登録**」→ 全件再インデックス
3. 類似検索で結果が返ることを確認

## STEP 8: 旧テナントドキュメント削除

STEP 6・7 が問題なければ、Firestore の**生キーID の旧ドキュメントを削除**
（サーバーログに `legacy plaintext-key tenant doc used` 警告が出なくなればOK）。

## 2社目テナントを追加するとき

1. 新しいAPIキーを発行（ランダム32文字以上）
2. `echo -n "新キー" | sha256sum` → ハッシュIDで `tenants` にドキュメント作成
   （tenantId = 新社の kintone サブドメイン, active = true）
3. 新社の kintone にプラグイン導入・設定（同じ Base URL＋新キー）
4. 注意: kintone 接続情報（KINTONE_BASE_URL/API_TOKEN）は現状サーバーに1組のみ。
   **別 kintone 環境の2社目を入れる前に Phase 3（テナント別 kintone 接続）が必要。**
