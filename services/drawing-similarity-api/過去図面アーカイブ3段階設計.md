# 過去図面アーカイブ：3段階設計

最終更新: 2026-07-09（下記「実装済み」を参照。3段階案は将来サムネイル表示が必要になった場合の参照用として残す）

## 実装済み（第一弾・2026-07-09）

個人開発の運用コストを踏まえ、当初の3段階案（台帳アプリ＋zip添付＋サムネイル表示）は採用せず、大幅に簡素化した最小構成でリリースした。

- **①軽量管理版相当のテキスト検索のみ**。サムネイルは生成・保存しない（Qdrantのpayloadにも持たない）
- **台帳アプリ・zip添付は不採用**。kintoneへの書き込みは一切行わない（`doc_type: 'archive'` として Qdrant にのみ登録）
- フォルダ入力は `<input type="file" webkitdirectory>` によるローカルの「フォルダ選択」のみ
- 実装箇所: `server.js` の `POST /archive-index`（新設）、`/index-status`・`/field-values` からのアーカイブ点除外、`/similar` の `docType` 分岐。`plugin.js` の `openArchiveIngestModal`（フォルダ選択→進捗→完了）、検索結果カードのアーカイブ表示分岐（汎用アイコン＋バッジ、kintoneリンクなし）
- サムネイル表示が将来必要になった場合、`archive_rel_path` は既にQdrant payloadに保存済みのため、再取込なしで表示層のみ追加すれば良い（下記②への移行と同等）

> **訂正（第二弾で判明）**：上記「NAS・Google Drive for Desktop等の同期フォルダもOSレベルで通常のローカルフォルダと同じに扱える」という想定は誤りだった。Google Drive for Desktopのオンデマンド同期（ストリーミング）モードでは、ローカルに実体がまだ無い「プレースホルダーファイル」をブラウザのFile APIが読めず `NotFoundError` になる。OS+ブラウザの構造的な制約で、実装での回避は不可能。対応は下記「実装済み（第二弾）」を参照。

## 実装済み（第二弾・2026-07-09）Google Drive連携（取り込みのみ）

ローカルフォルダ版のNotFoundError問題を受け、Google Driveに一本化してOAuth経由でDrive APIから直接ファイルを取得する方式を追加した。検討の結果、以下を採用：

- **`drive.file`スコープのみ使用**。フォルダ選択に対する永続的な読み取り権限（`drive.readonly`）は「制限付きスコープ」に該当し、有償のCASA Tier 2セキュリティ監査（$3000程度〜）が必須になるため不採用。取り込みのたびに毎回ファイルを選び直す運用（ローカルフォルダ版の「再取込で拾う」運用と同じ考え方）を受け入れた
- **サムネイル表示は今回もスコープ外**。検索結果を見る時点（取り込みの数週間後の可能性がある）でDriveに再アクセスするにはリフレッシュトークンの永続化（Firestore書き込み・新規エンドポイント・Googleの本番公開審査）が必要になり、実装量が大きく増えるため見送った。取り込みのみなら、ブラウザが直接取得する一時アクセストークン（有効期限1時間）だけで完結し、サーバー側の永続化は一切不要
- **マルチテナントのオリジン問題への対応**：Google Identity Servicesは呼び出し元ページのオリジンを事前登録する必要があり、kintoneの顧客サブドメインを個別登録するのは非現実的。そのため、OAuth同意とPicker選択は当方Cloud Run自身がホストする固定オリジンのポップアップページ（`GET /google/oauth/popup`）内で完結させ、結果を `postMessage` でプラグインに返す設計にした
- 実装箇所: `server.js` の `GET /google/oauth/popup`（新設、GSI+Picker読み込み用の固定HTML）、`/archive-index` への `driveFileId`→`drive_file_id` フィールド追加。`plugin.js` の `openArchiveIngestModal`（「Googleドライブから選択」ボタン追加）、`openGoogleDrivePicker`（ポップアップ制御）、`runArchiveIndex`（ローカルFile／Drive参照の両対応）、`fetchDriveFileBase64`
- 運用: Google Cloud ConsoleでOAuthクライアント（`drive.file`スコープ）を作成し、Authorized JavaScript originsに当方Cloud Runドメイン1つだけ登録する。テスト中はGoogleアカウント最大100件まで、本番展開前に標準審査（非機微スコープのためCASA監査は不要）が必要

## 前提（全段階で共通）

- **原本PDFは客先（NAS/Box等）から動かさない**。当方が保持するのはベクトル＋検索用テキストのみ
- 「図面と見られるもの」（サムネイル）は**客先 kintone に保存**する
- 新規図面の運用は現行フロー（1図面=1レコード＋原本添付）のままで、本設計は**過去図面の取り込み形態**のメニュー
- 取り込みフロー: フォルダを drag & drop → サーバーで OCR＋埋め込み（`doc_type: 'archive'`）→ サムネ生成 → kintone 台帳へ書き込み
- どの版でも、検索でヒットした重要図面は既存の「この図面を登録」で**個別レコードに昇格**できる
- 災害復旧: Qdrant 全損時も客先 kintone の台帳（metadata.json＋サムネ）から**再OCR不要で完全再構築**できる
  （埋め込みは 960px サムネから再生成可能。DINOv2 の入力解像度的に原本と実質同品質）

## 3段階の定義

| | ① 軽量管理版 | ② 軽量サムネ版 | ③ 全点管理版 |
|---|---|---|---|
| 台帳の形 | バッチレコード（1レコード=1袋） | バッチレコード（同左） | **1図面=1レコード** |
| 添付 | metadata.json＋保全用サムネzip（〜1000枚/袋・無圧縮） | metadata.json＋表示用サムネzip（**100〜200枚/袋**・無圧縮） | サムネPNG個別添付（原本添付はオプション） |
| 検索結果の表示 | 図番・品名・ファイル名・パス（テキストのみ） | 上記＋**サムネイル表示**（zip部分読み出し＋Cache API） | 上記＋サムネ、**kintoneレコードへのリンク** |
| kintone容量（1万枚） | 約0.5〜1GB | 約0.5〜1GB | サムネのみ約1GB／原本添付なら10〜20GB |
| 取り込みAPI消費（1万枚） | **約15〜30回** | 約60〜120回 | **1万回超 → 日次上限(目安1万/日/アプリ)に抵触。分割取り込み必須** |
| 初回取り込み時間 | 最短 | 短い | 数日に分割（スロットリング） |
| 図面単位の権限・コメント・編集 | 不可 | 不可 | **kintoneフル機能** |
| 想定規模・客層 | 数万枚級の大量アーカイブ／とにかく検索したい | 数千〜数万枚／見た目で判断したい（**推奨デフォルト**） | 〜数千枚／図面を資産管理したい |

## 技術要点

### 無圧縮zip＋クライアント部分読み出し（①②）
- PNG は圧縮済みのため zip は STORE（無圧縮）で作る → 目次（セントラルディレクトリ）を読んで**必要な1枚だけ slice** する処理が素のJSで書ける（ライブラリ不要）
- ②の袋サイズは表示レイテンシで決まる: 100〜200枚（5〜20MB）。初回検索のみバンドル取得の待ち、以降は Cache API から即時
- ①の袋は表示しないため 1000枚/袋で可（保全専用）

### 台帳レコード（①②共通）
```
図面アーカイブ台帳アプリ（1レコード = 1バンドル）
├ フィールド: バッチ名 / 取り込み日 / 件数 / 対象フォルダパス / 版種別
└ 添付: thumbs_NNNN.zip + metadata_NNNN.json（図番・品名・材質・元パス一覧）
```
- Qdrant payload には bundleRecordId＋zip内エントリ名を保持（②の表示用）

### ③の取り込み
- レコード作成は100件/リクエストでバッチ化、**添付アップロードは1枚1リクエスト**のため日次上限を跨ぐ分割実行のUI（進捗保存・再開）が必要
- 既存の一括登録UI（進捗バー・キャンセル）を流用

## 段階間の移行

- **①→②**: サーバー再処理不要。①の保全zipを 100〜200枚単位に再パッケージして台帳を差し替えるだけ（サムネは生成済み）
- **②→③**: 台帳の metadata.json＋zip から個別レコードを生成するバッチ（日次上限に合わせ分割）
- **①②→③（部分昇格）**: 検索ヒット時の「この図面を登録」で1枚ずつ（実装済み機能の流用）

## 料金メニューとの対応（PlugBits SaaS 化時）

3段階はそのままプラン名になり得る（例: アーカイブ検索 Lite / Standard / Pro）。
技術的な差分が「袋の粒度と台帳の形」だけなので、プラン変更＝移行バッチ実行で完結する。
