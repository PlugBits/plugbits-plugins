# Factory Tools - 図面登録機能 設計＋実装メモ

## 目的
図面登録を単一の入口にする。1回の登録操作で、
kintone製品マスターのレコード作成/更新と、Qdrantへのベクトル登録を同時に行う。
「検索のために登録する」のではなく「登録したら検索もできる」状態を作る。

## 最終目標
汎用SaaSとしてプラグイン化（PlugBits）。
ただし段階的に進める。まずは自社環境で動く版（Phase 1）を完成させる。

---

## データモデル（確定）
- 図番 = 製品マスターの主キー（重複禁止フィールド）
- 1製品（図番） = 1レコード = 1図面 = Qdrant 1ベクトル（すべて1対1）
- 図面ファイルは製品マスターアプリ内に添付（別アプリに分けない）
- 図面は差し替え・上書き運用（複数枚の追記はしない）
- **kintoneのレコードID = QdrantのポイントID**（紐付けキー）

## kintone製品マスター（運用中）
フィールドと型:
- 図番      → 文字列1行・重複禁止（主キー）
- 品名      → 文字列1行
- 加工方法  → 複数選択（配列で渡す）
- タグ      → 文字列1行・カンマつなぎ
- 図面ファイル → 添付ファイル

Phase 1ではフィールドコード直書きでよい。
Phase 2でプラグイン設定画面からのマッピングに置き換える。

---

## 登録フロー
1. 図面ドロップ
2. Cloud Run呼び出し
   - PDF→画像化
   - 右下クロップ → OCR（品名・図番候補を抽出）
   - DINOv2でベクトル化
3. Shadow DOMモーダルでユーザー確認・調整
   - OCR候補を「図番」「品名」にタップで振り分け（様式バラバラ前提、決め打ちしない）
   - 客先選択（様式学習の起点、Phase後半）
   - 加工方法（複数選択）・タグ（カンマつなぎ）を選択（固定選択肢、自由入力禁止）
4. 登録時に2箇所へ書き込み
   - kintone REST API: 図番で既存チェック → 上書き or 新規作成
   - Qdrant: kintone IDをポイントIDにしてupsert（差し替え時もベクトル再登録）

## 重要な実装ルール
- 図面差し替え時は **必ずベクトルも再登録**（古いベクトルが残ると検索が狂う）。
  同一IDでupsertすればQdrant側は自動上書き。
- エラー整合性: kintone成功→Qdrant失敗がありうる。
  Phase 1はユーザーに再試行させる簡易対応でよい。
  将来はkintoneに「未インデックス」フラグ＋バッチ再登録。
- UIはShadow DOM（`attachShadow({ mode: 'closed' })` + `ReactDOM.createRoot`）。
  kintoneのDOMを汚さない（既存GalleryKit等と同方針）。
- 既に「Cloud Run投げる＋タグ付け」プラグインが存在する。
  analyze/タグ付けは既存処理と重複しうる。既存があればそれを使い、
  registerDrawing() だけを足すのが綺麗。エンドポイント名（/analyze, /index）は
  既存サーバーの実パス・レスポンス形式に合わせて調整すること。

## 類似検索エンジン（実装済み・精度確定）
- モデル: dinov2-base, FLOOR=0.87
- スコア = DINOv2ベクトル類似度（メイン）+ Huモーメント + タグ加点
- 評価: Hit@1 82.6% / Hit@5 91.3% / Hit@10 95.7%（製品レベル達成済み）
- 精度追求はクローズ。残る失敗（PALLET FORK等）はUI（フォールバック）で対処。
- 生成AI（LLM）は未使用。DINOv2は推論のみ（学習しない・外部送信なし）。

## 汎用化（Phase 2以降）
- フィールド名: プラグイン設定でマッピング（直書き禁止）
- Cloud Run URL: 設定で入力
- マルチテナント: Qdrant payloadに tenant_id（= kintoneサブドメイン）を持たせフィルタ
- 認証/課金: Stripe + Cloudflare Workers + KV と連携（契約者のみCloud Run利用可）

## 開発フェーズ
- Phase 1: 自社環境で動く版（フィールド直書き）← 今ここ
- Phase 2: フィールドマッピング設定画面
- Phase 3: マルチテナント化（tenant_id分離・認証・課金連携）
- Phase 4: プラグインパッケージ化・配布
